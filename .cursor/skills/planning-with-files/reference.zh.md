# 参考：Manus 上下文工程原则

此技能基于 Manus 的上下文工程原则，Manus 是一家 AI 代理公司，于 2025 年 12 月被 Meta 以 20 亿美元收购。

## 6 条 Manus 原则

### 1. 文件系统作为外部内存

> "Markdown 是我在磁盘上的'工作记忆'。"

**问题：** 上下文窗口有限制。将所有内容塞入上下文会降低性能并增加成本。

**解决方案：** 将文件系统视为无限内存：
- 将大型内容存储在文件中
- 在上下文中只保留路径
- 代理可以在需要时"查找"信息
- 压缩必须是可逆的

### 2. 通过重复进行注意力操作

**问题：** 在约 50 个工具调用后，模型会忘记原始目标（"中间迷失"效应）。

**解决方案：** 保持一个 `task_plan.md` 文件，在整个执行过程中重新阅读：
```
上下文开始：[原始目标 - 很远，被遗忘]
...许多工具调用...
上下文结束：[最近阅读的 task_plan.md - 获得注意力！]
```

通过在每次决策前阅读计划文件，目标会出现在注意力窗口中。

### 3. 保留失败痕迹

> "错误恢复是真正代理行为最清晰的信号之一。"

**问题：** 本能会隐藏错误，静默重试。这会浪费 token 并失去学习机会。

**解决方案：** 在计划文件中保留失败的操作：
```markdown
## 遇到的错误
- [2025-01-03] FileNotFoundError: config.json 未找到 → 创建了默认配置
- [2025-01-03] API 超时 → 使用指数退避重试，成功
```

模型在看到失败时会更新其内部理解。

### 4. 避免少样本过拟合

> "一致性滋生脆弱性。"

**问题：** 重复的动作-观察对会导致漂移和幻觉。

**解决方案：** 引入受控变化：
- 稍微改变措辞
- 不要盲目复制粘贴模式
- 在重复任务上重新校准

### 5. 用于缓存优化的稳定前缀

**问题：** 代理是输入密集型的（100:1 比例）。每个 token 都要花钱。

**解决方案：** 为缓存命中而结构化：
- 将静态内容放在最前面
- 仅追加上下文（永远不修改历史）
- 一致的序列化

### 6. 仅追加上下文

**问题：** 修改先前的消息会使 KV 缓存失效。

**解决方案：** 永远不要修改先前的消息。始终追加新信息。

## 代理循环

Manus 在连续循环中运行：

```
1. 分析 → 2. 思考 → 3. 选择工具 → 4. 执行 → 5. 观察 → 6. 迭代 → 7. 交付
```

### 循环中的文件操作：

| 操作 | 何时使用 |
|------|----------|
| `write` | 新文件或完全重写 |
| `append` | 增量添加部分 |
| `edit` | 更新特定部分（复选框、状态） |
| `read` | 决策前审查 |
