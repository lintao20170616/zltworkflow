# Design: 低代码页面构建工具

## Context

当前团队开发配置列表查询页面和表单页面时，存在大量重复性代码编写工作，开发效率低下。传统页面开发方式需要编写大量重复的HTML/CSS/JS代码，配置逻辑相似度高但手工编写耗时，且难以统一维护。

项目已具备Vue 3 + Element Plus技术栈，可以基于现有技术栈构建低代码页面构建工具。第一版主要面向本地使用，无需后端服务，使用浏览器本地存储。

## Goals / Non-Goals

### Goals
- 实现列表查询页面和表单页面的可视化配置
- 支持PC端和移动端双端预览
- 输出标准JSON配置供渲染组件使用
- 配置效率提升60%以上
- 支持不少于20种常用组件
- JSON配置生成时间<500ms

### Non-Goals
- 不实现用户权限管理（第一版）
- 不实现版本管理与历史回滚（第一版）
- 不实现云端存储与多端同步（第一版）
- 不实现自定义组件扩展（第一版）
- 不实现代码生成（仅生成JSON配置）

## Decisions

### Decision: 使用Vue 3 + Element Plus技术栈
**原因**：
- 项目已使用Vue 3 + Element Plus，技术栈统一
- Element Plus组件丰富，适合快速搭建管理类页面
- Vue 3 Composition API更适合复杂状态管理
- 团队熟悉Vue 3技术栈

**替代方案**：
- React + Ant Design：需要引入新的技术栈，增加学习成本
- 原生HTML/CSS/JS：开发效率低，维护成本高

### Decision: 使用Pinia进行状态管理
**原因**：
- 项目已集成Pinia
- Pinia相比Vuex更轻量，API更简洁
- 支持TypeScript类型推断
- 适合单页面复杂状态管理

**替代方案**：
- Vuex：更重量级，API较复杂
- 组件内状态：无法满足跨组件状态共享需求

### Decision: 使用vue-draggable-plus实现拖拽
**原因**：
- 专为Vue 3设计，API简洁，集成度高
- 支持画布内组件排序和移动（核心需求）
- 支持嵌套列表拖拽（嵌套组件场景）
- 提供拖拽动画和视觉反馈，用户体验好
- 性能优化，支持大列表和虚拟滚动
- 移动端支持良好
- 基于Sortable.js，功能完善且稳定

**替代方案**：
- 原生HTML5拖拽API：不支持列表内排序，实现复杂，移动端兼容性差
- Sortable.js：Vue 3组件封装过时，需要手动集成
- 自定义拖拽实现：开发成本高，稳定性差，难以实现复杂交互

### Decision: 使用LocalStorage进行本地存储
**原因**：
- 第一版仅支持本地使用，无需服务端
- LocalStorage简单易用，无需额外配置
- 存储容量足够（5-10MB）
- 支持JSON序列化/反序列化

**替代方案**：
- IndexedDB：API复杂，存储容量大但第一版不需要
- 后端数据库：需要后端服务，第一版不实现

### Decision: JSON配置Schema设计
**方案**：
```json
{
  "version": "1.0.0",
  "page": {
    "title": "页面标题",
    "layout": "container"
  },
  "components": [
    {
      "id": "comp_001",
      "type": "el-form",
      "props": {
        "labelWidth": "100px"
      },
      "children": [...]
    }
  ]
}
```

**原因**：
- 结构清晰，易于解析
- 支持嵌套组件
- 便于版本管理（version字段）
- 易于扩展

### Decision: 布局组件嵌套设计
**方案**：
- 在`ComponentSchema`中添加`canNest`字段，标识组件是否支持嵌套子组件
- 布局组件（`el-row`、`el-col`、`el-container`、`el-card`、`el-form`、`el-form-item`、`el-space`、`el-badge`）标记为`canNest: true`
- 使用`vue-draggable-plus`的`group`功能实现嵌套拖拽
- 嵌套容器内支持组件排序和移动
- 支持从组件库直接拖拽到布局组件内部
- 嵌套组件有内边距（padding），视觉上体现层级关系

**实现细节**：
- `ComponentItem`组件递归渲染，检测`canNest`属性
- 可嵌套组件内部使用`VueDraggable`包裹子组件列表
- 空容器显示占位提示，引导用户拖拽组件
- 拖拽时使用`group="components"`确保跨容器拖拽
- 嵌套容器添加12px内边距，子组件视觉上比父组件小
- 嵌套组件有正确的z-index层级，选中效果能完全覆盖

**el-row特殊处理**：
- 编辑器中使用`nest-container-row`类，应用flex布局
- 子组件保持自然宽度，不强制100%宽度
- gutter属性通过CSS变量`--el-row-gutter`传递和应用
- 预览渲染器中，el-row的直接子组件直接渲染，避免多余的`lowcode-renderer`包装

**原因**：
- 支持复杂页面布局结构（如栅格系统、卡片嵌套）
- 提供直观的拖拽体验，用户可直接将组件拖入布局容器
- 嵌套组件支持独立排序，不影响外层组件
- 符合Element Plus组件的实际使用场景
- 所见即所得，编辑器效果与预览效果一致

**替代方案**：
- 仅支持扁平结构：无法实现复杂布局，限制页面设计能力
- 手动配置children：用户体验差，不符合可视化编辑理念

### Decision: 渲染器组件独立设计
**方案**：
- 渲染器组件独立于编辑器
- 可嵌入现有业务系统
- 接收JSON配置和动态数据
- 支持事件回调
- el-row/el-col特殊处理：直接渲染子组件，避免多余的lowcode-renderer包装

**实现细节**：
- `LowcodeRenderer.vue`负责预览和最终渲染
- 对于el-row和el-col，子组件直接作为其子元素渲染，不使用lowcode-renderer包裹
- 动态设置CSS变量支持gutter等属性
- 支持递归渲染嵌套组件

**原因**：
- 编辑器与渲染器解耦
- 渲染器可独立使用和测试
- 便于集成到现有系统
- 支持多环境使用
- 确保布局组件（如el-row）的flex布局正常工作

## Risks / Trade-offs

### 风险1: 组件属性配置复杂度高
**缓解措施**：
- 优先实现核心属性，逐步扩展
- 使用属性Schema统一管理
- 提供属性配置模板
- 支持属性分组和搜索

### 风险2: 动态渲染性能问题
**缓解措施**：
- 使用Vue 3的响应式系统优化
- 组件懒加载
- 虚拟滚动（如需要）
- 性能监控和优化

### 风险3: 移动端兼容性问题
**缓解措施**：
- 提前测试主流移动设备浏览器
- 使用Element Plus的响应式组件
- 提供移动端预览功能
- 兼容性测试清单

### 风险4: 数据绑定逻辑复杂
**缓解措施**：
- 设计清晰的数据绑定规范
- 提供数据绑定示例和文档
- 支持v-model双向绑定
- 简化事件处理机制

### 权衡: 同步 vs 异步配置更新
- **同步**：实时反馈，用户体验好，但可能影响性能
- **异步**：性能好，但反馈延迟

**选择**：使用同步更新，通过防抖优化性能

### 权衡: 完整功能 vs 快速上线
- **完整功能**：功能完善，但开发周期长
- **快速上线**：先实现核心功能，后续迭代

**选择**：第一版实现核心功能，辅助功能后续迭代

## Migration Plan

### 实施步骤
1. 项目初始化和依赖安装
2. 状态管理和数据结构设计
3. 组件库模块开发
4. 画布编辑模块开发
5. 属性配置模块开发
6. 预览模块开发
7. JSON配置生成模块开发
8. 页面渲染器模块开发
9. 布局容器组件开发
10. 模板管理功能开发
11. UI优化和体验提升
12. 测试和文档完善

### 回滚方案
- 如果低代码工具出现问题，可以：
  - 禁用编辑器入口（前端隐藏）
  - 不影响现有业务系统
  - 渲染器组件独立，可单独使用

### 集成方案
- 编辑器作为现有应用的一个功能模块，通过路由访问（`/lowcode/editor`）
- 渲染器组件作为公共组件，供业务系统使用
- JSON配置通过文件或API传递
- 编辑器页面集成在现有Layout布局中

### Decision: 自定义样式编辑器设计
**方案**：
- 使用键值对输入方式替代JSON文本输入
- 提供添加、删除样式项的功能
- 实时更新样式到组件
- 空状态提示引导用户

**实现细节**：
- `StyleEditor.vue`组件提供键值对输入界面
- 属性名和属性值分别输入（如：width -> 100px）
- 支持实时更新，无需手动解析JSON
- 空状态显示提示信息

**原因**：
- 比JSON文本输入更直观易用
- 降低用户学习成本
- 减少JSON格式错误
- 提供更好的用户体验

**替代方案**：
- JSON文本输入：需要用户了解JSON格式，容易出错
- 可视化样式编辑器：开发成本高，第一版暂不实现
