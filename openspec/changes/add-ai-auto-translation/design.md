# Design: AI自动翻译功能

## Context
当前翻译平台需要人工逐条翻译，效率较低。系统已集成Ollama服务用于聊天机器人功能，可以复用该基础设施实现AI自动翻译。

## Goals / Non-Goals

### Goals
- 实现单个翻译内容的AI自动翻译
- 实现批量翻译任务的AI自动翻译
- 复用现有Ollama服务基础设施
- 提供良好的错误处理和用户体验

### Non-Goals
- 不实现新的AI服务集成（复用Ollama）
- 不实现翻译质量评估和评分
- 不实现翻译记忆库（TM）功能
- 不实现多引擎翻译对比

## Decisions

### Decision: 使用Ollama服务进行翻译
**原因**：
- 系统已具备Ollama服务集成，无需额外依赖
- Ollama支持本地部署，数据安全可控
- 已有完善的错误处理和降级机制

**替代方案**：
- OpenAI API：需要API密钥，成本较高
- Google Translate API：需要API密钥，有使用限制
- 其他第三方翻译服务：增加外部依赖

### Decision: 翻译prompt设计
**方案**：
使用结构化的prompt模板，包含：
- 明确的翻译任务说明
- 源语言和目标语言信息
- 源文本内容
- 翻译要求（保持原意、专业术语等）

**原因**：
- 结构化prompt能提供更好的翻译质量
- 可配置的模板便于后续优化
- 支持多语言对翻译

### Decision: 翻译状态管理
**方案**：
- AI翻译成功后，状态自动更新为"已完成"（status = 3）
- translatorId标记为系统用户ID或特殊值（如0或-1）

**原因**：
- 保持与人工翻译一致的状态流转
- 便于区分AI翻译和人工翻译
- 支持后续的审核流程

### Decision: 批量翻译实现方式
**方案**：
- 同步逐个翻译，实时更新进度
- 记录每个翻译的成功/失败状态
- 返回整体统计信息

**替代方案**：
- 异步批量翻译：实现复杂，需要任务队列
- 并行翻译：可能对Ollama服务造成压力

**原因**：
- 实现简单，易于调试
- 实时反馈进度
- 避免对Ollama服务造成过大压力

## Risks / Trade-offs

### 风险1: Ollama服务不可用
**缓解措施**：
- 实现完善的错误处理
- 提供友好的错误提示
- 支持重试机制
- 翻译状态不更新，保持为"待翻译"

### 风险2: 翻译质量不稳定
**缓解措施**：
- 设计优化的prompt模板
- 支持prompt配置和调优
- 提供人工审核流程
- 用户可手动修改AI翻译结果

### 风险3: 批量翻译性能问题
**缓解措施**：
- 逐个翻译，避免并发压力
- 记录翻译耗时，便于优化
- 支持分批处理（未来可扩展）

### 权衡: 同步 vs 异步批量翻译
- **同步**：实现简单，实时反馈，但可能耗时较长
- **异步**：用户体验好，但实现复杂，需要任务队列

**选择**：先实现同步方式，后续根据需求考虑异步优化

## Migration Plan

### 实施步骤
1. 后端实现AI翻译服务方法
2. 后端实现API接口
3. 前端实现UI和API调用
4. 测试和优化
5. 部署和监控

### 回滚方案
- 如果AI翻译功能出现问题，可以：
  - 禁用AI翻译按钮（前端隐藏）
  - 配置Ollama服务不可用时的降级处理
  - 不影响现有的人工翻译流程
