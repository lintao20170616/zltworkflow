## 1. 项目现状分析

### 1.1 已完成模块（95%+）


| 模块        | 完成度  | 说明                            |
| --------- | ---- | ----------------------------- |
| 项目管理      | 95%  | 新建、编辑、删除、复制项目，基于 localStorage |
| 属性配置面板    | 90%  | 动态属性展示、多种编辑器、实时更新             |
| 样式编辑器     | 100% | 键值对样式编辑、实时同步                  |
| 状态管理      | 95%  | Pinia store，支持组件增删改、撤销重做、导入导出 |
| Schema 定义 | 95%  | 20+ Element Plus 组件 Schema 定义 |


### 1.2 未完成核心模块（0%）


| 模块     | 完成度 | 说明                   |
| ------ | --- | -------------------- |
| 画布渲染   | 0%  | 无法接收拖拽、无法渲染组件、无法选中编辑 |
| 组件库分类  | 80% | 仅表单组件，硬编码，未复用 Schema |
| 事件系统   | 0%  | 无事件绑定和执行机制           |
| 数据绑定   | 0%  | 无表单数据绑定、无接口调用        |
| 生产页面生成 | 0%  | 无法将 JSON 配置转换为生产页面   |


### 1.3 核心问题诊断

**问题一：有脑无手**

- 基础设施完备（状态管理、属性配置、Schema 定义）
- 但画布渲染缺失，无法形成"拖拽 → 渲染 → 选中 → 配置"的闭环

**问题二：架构不一致**

- 当前 store 使用扁平数组结构
- 需要改造为递归树结构以支持嵌套

**问题三：数据源缺失**

- 组件拖入后无默认数据，用户看到空白组件
- 无数据绑定机制，无法连接真实业务

## 2. 实施目标

### 2.1 核心目标

1. **打通核心编辑流程**：实现画布拖拽、渲染、选中的完整闭环
2. **支持嵌套布局**：实现递归树结构和容器组件嵌套
3. **开箱即用**：组件拖入后有默认数据和演示效果
4. **事件绑定**：支持组件事件配置和执行（先实现 console.log）
5. **生成生产页面**：将 JSON 配置转换为可访问的生产页面

### 2.2 量化指标

- 支持 20+ 可用组件
- 支持容器组件嵌套（el-card、el-tabs、el-form 等）
- 组件库按 4 大类分组（基础、表单、展示、业务模版）
- 拖拽插入支持 3 种位置（前、后、容器内）
- 页面构建操作响应时间 < 200ms
- 生产页面加载时间 < 1s

## 3. 技术方案总览

### 3.1 整体架构

```
┌─────────────────────────────────────────┐
│           编辑器环境（Editor）             │
│  - 组件库 + 画布 + 属性面板 + 样式编辑器    │
└──────────────┬──────────────────────────┘
               │ JSON 配置
┌──────────────▼──────────────────────────┐
│       状态管理层（Pinia Store）          │
│  - 递归树结构 + 事件系统 + 数据绑定        │
└──────────────┬──────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│      数据存储层（localStorage）           │
│  - 项目配置 + 组件 Schema                │
└─────────────────────────────────────────┘
               │
┌──────────────▼──────────────────────────┐
│        生产环境（Runtime Renderer）       │
│  - 页面入口 + 组件渲染器 + 事件执行        │
└─────────────────────────────────────────┘
```

### 3.2 核心技术决策


| 决策项       | 方案                | 理由               |
| --------- | ----------------- | ---------------- |
| 数据结构      | 递归树               | 支持嵌套布局，符合 DOM 结构 |
| 渲染模式      | 递归组件 + 包装器        | 支持嵌套和选中交互        |
| 拖拽模式      | 精细模式（3 种位置）       | 用户体验好，支持精确排序     |
| 包裹层处理     | 编辑/预览双模式 + CSS 补偿 | 编辑时可用，预览时准确      |
| 属性 Schema | 自动生成 + 人工增强       | 平衡效率与质量          |
| 事件处理      | 先实现 log，预留 API    | 快速验证，后续扩展        |
| 生产渲染      | 运行时动态渲染           | 灵活性强，配置变更无需重新部署  |

## 5. 技术实施细节

### 5.1 Store 数据结构

```typescript
// store/lowcode.ts
interface Component {
  id: string
  type: string
  category: 'basic' | 'form' | 'display' | 'template'
  props: Record<string, any>
  style?: Record<string, any>
  children?: Component[]
  events?: {
    [eventName: string]: EventConfig[]
  }
}

interface LowcodeState {
  rootComponent: Component | null
  selectedComponentId: string | null
  previewMode: boolean
  history: {
    past: Component[][]
    present: Component | null
    future: Component[][]
  }
}
```

### 5.2 组件渲染流程

```
用户拖拽组件
  → handleDragStart (设置 dataTransfer)
  → handleDragOver (计算插入位置)
  → handleDrop (调用 store.addComponent)
  → store.createComponent (合并 defaultConfig 和 mockData)
  → 更新 rootComponent
  → ComponentWrapper 重新渲染
  → 显示组件和默认数据
```

### 5.3 事件执行流程

```
用户点击按钮
  → handleNativeClick (绑定 @click)
  → executeEventHandlers('click', context)
  → handleLogEvent (解析 {{form.username}})
  → console.log('[Event Log]', message, data)
```

### 5.4 生产页面渲染流程

```
访问 /page/:id
  → Viewer.vue 加载
  → loadPageConfig(pageId)
  → 解析 JSON 配置
  → ComponentRenderer 递归渲染
  → 执行用户事件（如提交表单）
```

## 7. 风险管理

### 7.1 技术风险


| 风险项        | 影响  | 概率  | 应对措施                         |
| ---------- | --- | --- | ---------------------------- |
| 递归渲染性能问题   | 高   | 中   | 组件数量 >100 时，考虑虚拟滚动           |
| 拖拽插入点计算不准确 | 中   | 中   | 充分测试不同组件、不同尺寸                |
| 容器组件边界判断   | 中   | 低   | 明确 `CONTAINER_COMPONENTS` 列表 |
| 撤销重做内存占用   | 低   | 低   | 限制历史步数 50 步                  |


### 7.2 进度风险


| 风险项          | 影响  | 概率  | 应对措施               |
| ------------ | --- | --- | ------------------ |
| 阶段一复杂度超预期    | 高   | 中   | 先实现简单拖拽，精细模式后续优化   |
| Schema 统一工作量 | 中   | 中   | 先统一表单类，逐步扩展        |
| 事件系统复杂度      | 中   | 低   | 先只实现 log，预留 API 接口 |


### 7.3 应对策略

1. **分阶段交付**：每阶段结束后进行演示和验收，确保方向正确
2. **MVP 优先**：先实现最小可用版本，后续逐步优化
3. **持续集成**：每完成一个模块立即测试，避免累积风险
